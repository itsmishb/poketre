# ポケモンカード販売業者向け カード管理システム — 調査と推奨設計

## 1. はじめに

本ドキュメントは、**スキャナー→OCR→システム登録**を軸に、**在庫管理（フリマ・オークション・自社EC）**・**Shopify連携**・**棚管理**・**相場取得**・**ダッシュボード**までをカバーする「カード販売業者向け管理システム」について、**既存の世の中のシステムを調査した結果**と、**御社向けの推奨設計**をまとめたものです。

共有いただいた構想（PDF「カード管理システム概要」および過去の設計メモ）を土台としつつ、**「管理システム本体を基幹」「Shopifyは出品チャネルの1つ」「フリマ・オークションは掲載・成約の管理対象」**という前提で整理しています。

## 2. 既存システムの調査結果（参考事例・URL一覧）

### 2.1 分類の考え方

| 類型 | 特徴 | 御社との関係 |
|------|------|----------------|
| **A. カード/TCG特化** | カード名寄せ・セット・相場・出品に強い | 機能参考・競合イメージ |
| **B. 汎用POS/在庫/OMS** | 在庫・受注・出荷・ダッシュボードが強い | 業務フロー・状態設計の参考 |
| **C. Shopify中心** | EC・決済・受注が強い | 連携先として設計に組み込む |
| **D. スキャン/コレクション** | OCR・画像認識・価格表示 | 入力レイヤー・UXの参考 |

御社ケースでは **「カード特化の内部基幹」＋「Shopifyは販売チャネルの1つ」** が最適です。

## 3. 御社向けの推奨設計の整理

### 3.1 システムの位置づけ

- **管理システム本体** = 基幹（OCR登録・カード台帳・在庫・棚・掲載・成約・価格・ダッシュボード）
- **Shopify** = 自社ECとしてAPIで商品・在庫を同期する「販売チャネルの1つ」
- **ヤフオク・フリマ** = API出品はせず、「どこに何を何枚出しているか」「成約したか」を管理する対象

データの正本は**常に管理システム側**に置き、Shopifyで売れたら**注文を取り込み→在庫引当・出庫**する流れにします。

### 3.2 全体フロー（既存構想を活かした形）

```
スキャナー → 画像保存（例: Drive） → OCR/AI抽出
    → OCR_Staging（候補）
    → 人が確認・修正・OK/NG
    → CardCatalog / Inventory（Cards + 現物在庫）
    → StockMovements（差分で在庫管理）
    → Listings（チャネル別掲載）
        ├─ Shopify: API同期
        ├─ ヤフオク: 管理のみ
        └─ フリマ: 管理のみ
    → 成約・注文登録 → 在庫引当・出庫
    → 棚管理（StorageLocations）
    → 価格スナップショット（任意）
    → ダッシュボード
```

### 3.3 設計の重要方針（既存資料＋調査を踏まえた結論）

| 方針 | 説明 |
|------|------|
| **基幹はShopifyにしない** | 在庫・カードマスタ・棚は自社システムが正本。Shopifyは同期先。 |
| **在庫は差分台帳** | 在庫数は「StockMovements の qty_delta 合計」で算出。直接書き換えしない。 |
| **カード種別と現物を分ける** | CardCatalog（種別）と InventoryUnit / InventoryLot（現物・状態・枚数）を分離。高額は1枚単位、安価はロット単位のハイブリッドが現実的。 |
| **出品＝在庫引当** | 掲載時に RESERVED、成約で SOLD_PENDING→出庫で SHIPPED。複数チャネルで二重販売しないよう状態を明確に。 |
| **Listings はチャネル掲載台帳** | listing_mode: API_SYNC（Shopify） / MANUAL_MANAGED（ヤフオク・フリマ）。external_listing_id・sync_status・reserved_qty を持つ。 |
| **棚管理を独立で持つ** | StorageLocations（倉庫・ゾーン・棚・ボックス・スロット）と、在庫との紐づけで「どこに保管しているか」を管理。 |
| **価格は履歴で持つ** | PriceSnapshots で取得日時・ソース・価格種別（出品/成約/参考）を分けて保存。取得方法はAPI/手動/CSV/スクレイピングのどれでも差し込める設計に。 |
| **MVPはスプシ可、スキーマはDB前提** | シート名・項目は将来のDB移行を想定して設計する。 |

### 3.4 推奨テーブル（シート）構成（MVP〜本番）

- **Sets** … セット台帳（set_code, set_name_ja, series, release_date, total_cards, regulation_set）
- **CardCatalog（Cards）** … カード種別マスタ（serial_number, set_code, card_number, number_total, name_ja, regulation_mark, rarity, card_type, image_ref 等）
- **OCR_Staging** … OCR候補。確認・初期数量・状態・棚番・review_status・approved_inventory_type 等を追加推奨。
- **InventoryUnits** … 高額・個別管理用（card_catalog_id, condition_grade, storage_location_id, status, front_image_url 等）
- **InventoryLots** … 安価・複数枚ロット用（card_catalog_id, condition_grade, qty_on_hand, storage_location_id, status）
- **StockMovements** … 入出庫・調整・予約・解除・移動。target_type（UNIT/LOT）, movement_type（IN/OUT/ADJUST/RESERVE/RELEASE/TRANSFER）, ref_kind, ref_id
- **StorageLocations** … 棚番（warehouse, zone, shelf, rack, bin, slot, barcode）
- **ChannelListings（Listings）** … チャネル別掲載（channel, listing_mode, external_listing_id, sync_status, reserved_qty, status, list_qty, price, sold_at 等）
- **ChannelOrders** … 注文（channel, external_order_id, card_id/unit_id, qty, sold_price, ordered_at, import_status）
- **PriceSnapshots** … 相場履歴（card_catalog_id, source_name, source_type, price_type, price_value, fetched_at）
- **SyncJobs** … 連携実行ログ（Shopify同期など）
- **Operators** … 作業者（任意）

※ 既存の「Cards」「Listings」は名前を CardCatalog / ChannelListings に拡張するか、役割を上記のように明確にしたうえでそのままでも可です。

### 3.5 Shopify連携の役割

- **やること**: 商品作成/更新、在庫数更新、画像・価格反映、注文取得。
- **やらないこと**: 基幹在庫の正本、OCR登録、棚番、他チャネル在庫統制の主導。

連携用に **ChannelProducts**（channel, card_id, shopify_product_id, shopify_variant_id, sync_status, last_synced_at）があると運用しやすいです。

### 3.6 相場取得

- **PriceSnapshots** に source_type（API / SCRAPING / MANUAL / CSV）、price_type（ASK / SOLD / LOWEST / MEDIAN 等）を分けて保存。
- スクレイピングする場合は各サイトの利用規約と法的要件を確認し、**設計はスクレイピングに依存しない**形にしておくことを推奨します（後からAPIやCSVに差し替え可能にする）。

### 3.7 ダッシュボードで持つとよい指標

- **在庫**: 総在庫数・在庫金額・出品可能数・チャネル別出品数・棚別在庫・登録待ち件数
- **販売**: 日次/週次/月次売上・粗利・チャネル別売上・回転率・滞留在庫
- **業務**: OCR成功率・確認待ち数・連携エラー数・棚卸差異・二重販売リスク
- **価格**: 相場推移・自社価格との差・値付け見直し対象

## 4. 実装の順序（提案）

1. OCR_Staging の拡張（確認・初期数量・状態・棚番）
2. CardCatalog / InventoryLots / InventoryUnits の整理
3. StorageLocations の追加
4. ChannelListings（Listings）の拡張
5. Shopify連携（商品・在庫・注文取り込み）
6. PriceSnapshots と価格取得の差し込み口
7. ダッシュボード
8. 必要に応じてDB移行（PostgreSQL 等）
