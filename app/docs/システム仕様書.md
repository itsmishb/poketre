## 1. システム構成

### 1.1 全体アーキテクチャ

```
┌─────────────────────────────────────────────────────────────────────────┐
│ ユーザー（作業者・管理者）                                                    │
└─────────────────────────────────┬───────────────────────────────────────┘
                                  │
                                  ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ 管理システム（本システム）                                                    │
│ ・登録待ち確認UI  ・在庫/棚/掲載/価格 管理画面  ・ダッシュボード              │
│ ・AppSheet または 自前Web（MVP期はスプレッドシート連携）                          │
└─────────────────────────────────┬───────────────────────────────────────┘
                                  │ 読取・更新
                                  ▼
┌─────────────────────────────────────────────────────────────────────────┐
│ データストア（MVP: Google Sheets / 将来: RDB）                               │
│  Sets | CardCatalog | OCR_Staging | InventoryUnits | InventoryLots |     │
│  StockMovements | StorageLocations | ChannelListings | ChannelProducts |  │
│  ChannelOrders | PriceSnapshots | SyncJobs | （管理用追加列）                   │
└─────────────────────────────────┬───────────────────────────────────────┘
                                  │
        ┌─────────────────────────┼─────────────────────────┐
        │                         │                         │
        ▼                         ▼                         ▼
┌───────────────┐   ┌───────────────────────┐   ┌───────────────────────┐
│ 既存 Cloud Run │   │ Shopify（クライアントEC）  │   │ 価格取得（API/手動/CSV等） │
│ ・Drive Inbox │   │ ・商品・在庫・注文 API   │   │ ・PriceSnapshots 投入   │
│ ・Gemini OCR  │   │ ・本システムが同期元     │   │ ・差し替え可能設計       │
│ ・OCR_Staging │   └───────────────────────┘   └───────────────────────┘
│   追記のみ     │
└───────────────┘
```

- **本システム**: 管理画面とデータストアへの読み書き、Shopify/価格との連携を担当する。OCR の「書き込み」は行わない。
- **Cloud Run**: 既存のまま。Drive Inbox → Gemini → OCR_Staging 追記・画像フォルダ移動。本仕様の変更対象外。
- **データストア**: MVP 期は同一 Google スプレッドシート内の複数シート。将来は PostgreSQL 等へ移行可能なスキーマで設計する。

### 2.2 画面遷移（主要フロー）

- **登録フロー**: ホーム or メニュー → 登録待ち一覧(SCR-001) → 登録待ち詳細(SCR-002) → OK で正式登録 → カード種別一覧(SCR-003) or 在庫一覧(SCR-006) へ誘導可能。
- **在庫操作**: 在庫一覧(SCR-006) → 在庫詳細(SCR-007) で入出庫記録。
- **出品フロー**: 掲載一覧(SCR-009) → 掲載登録(SCR-010)。手動チャネルは成約時に SCR-010 で成約登録。
- **管理**: 棚番(SCR-008)、セット(SCR-005)、Shopify 設定(SCR-011)、価格(SCR-013/014)、ダッシュボード(SCR-015) はメニューから直接アクセス。

### 2.3 表示言語・ロケール

本システムは日本人利用者向けとする。画面に表示するラベル・メッセージ・日付・数値・通貨はすべて**日本語**および**日本向け形式**（YYYY年MM月DD日または YYYY/MM/DD、桁区切りカンマ、通貨は円）とする。要件定義書「3. 表示言語・ロケール」および UI 要件定義書に従う。

### 2.4 画面別仕様（主要画面）

#### SCR-001 登録待ち一覧

- **表示**: OCR_Staging のうち status＝「登録待ち」を一覧。ページングまたはスクロールで対応。
- **列**: サムネイル（image_url）、serial_number、name_ja、set_code、card_number_text、rarity、qty、confidence、review_status（本システム追加列）など。ソート・フィルタ（set_code、rarity、review_status）を用意する。
- **操作**: 行クリックで SCR-002 へ遷移。一括 OK/NG は任意（MVP では 1 件ずつでも可）。

#### SCR-002 登録待ち詳細・確認

- **表示**: 選択した 1 件の image_url（大きく表示）、OCR 抽出項目（serial_number、name_ja、set_code、card_number_text、rarity、card_type、poke_type、trainer_subtype、generation、PSA 情報、confidence、holo 等）を編集可能なフォームで表示。
- **本システム追加入力項目**: initial_qty（初期数量）、condition_grade（状態）、storage_location_id（棚番）、review_status（OK/NG/要再スキャン）、approved_inventory_type（UNIT or LOT）。intended_channels は任意。
- **操作**: 「OK（正式登録）」押下で F4 の正式登録処理を実行。「NG」「要再スキャン」は review_status および status を更新するのみ。戻るで SCR-001 へ。

#### SCR-006 在庫一覧

- **表示**: InventoryUnits と InventoryLots を一覧（またはタブ/フィルタで切替）。カード種別名・serial_number、状態、棚番、在庫数（Unit は 1、Lot は qty_on_hand）、status、保管場所名を表示。
- **検索・フィルタ**: card_catalog_id / serial_number、storage_location_id、status、condition_grade で絞り込み。
- **操作**: 行選択で SCR-007 へ。

#### SCR-007 在庫詳細・入出庫記録

- **表示**: 対象 Unit または Lot の詳細（カード種別、状態、棚番、status、画像、取得原価等）。紐づく StockMovements の履歴（日時、qty_delta、movement_type、ref_kind、notes）を表示。
- **操作**: 「入庫を記録」「出庫を記録」「調整」等のボタンで StockMovements に 1 行追加。qty_delta、movement_type、ref_kind、notes を入力。必要に応じて storage_location_id の変更（移動）も可能にする。

#### SCR-009 掲載一覧

- **表示**: ChannelListings を channel・status でフィルタ可能な一覧。listing_id、channel、対象カード（serial_number 等）、list_qty、price、status、sync_status（Shopify の場合）、published_at、ended_at 等を表示。
- **操作**: 新規掲載は SCR-010。成約登録（手動チャネル）は SCR-010 で該当掲載を開き status を SOLD に更新し在庫出庫を実行。

#### SCR-010 掲載登録・編集

- **新規掲載**: channel、対象在庫（target_type/target_id または card_catalog_id＋数量）、list_qty、price、listing_title、listing_description、listing_image_urls を入力。API_SYNC の場合は「Shopify に同期」で F14 を実行。MANUAL_MANAGED の場合は status を LISTED にし、在庫を RESERVE（StockMovements に RESERVE 記録）。
- **成約登録（手動）**: 該当掲載の status を SOLD に更新、sold_at・sold_price を入力。在庫を OUT（StockMovements に OUT、ref_kind=ORDER 等）。

#### SCR-015 ダッシュボード

- **在庫**: 総在庫数、在庫金額（評価額）、出品可能数、チャネル別出品数、棚別在庫数、登録待ち件数。期間は「今日」または指定なしで現在値。
- **販売**: 日次/週次/月次の売上、粗利、チャネル別売上、回転率、滞留在庫ランキング。集計期間を選択可能に。
- **業務**: OCR 確認待ち件数、連携エラー数（SyncJobs の直近失敗）、棚卸差異（任意）、二重販売リスク（掲載中だが在庫不足等の検知、任意）。
- **価格**: 相場推移（PriceSnapshots の時系列）、自社価格との差、値付け見直し対象。カード種別またはセットでフィルタ可能に。

## 4. 処理仕様

### 4.1 正式登録処理（F4）

**トリガー**: SCR-002 で「OK（正式登録）」押下。

**入力**: OCR_Staging の 1 行（stg_id で特定）。入力項目: initial_qty、initial_condition、storage_location_id、approved_inventory_type（UNIT or LOT）、修正後の serial_number / set_code / card_number / number_total / name_ja / rarity / card_type 等。

**処理手順**:

1. **CardCatalog の UPSERT**
   - card_id を生成: 例 `{set_code}-{card_number}-{rarity}`（重複時はサフィックス）。
   - serial_number は OCR の serial_number を利用（例: sv8_100/106）。
   - 既に同一 card_id が存在すれば更新しないか、必要項目のみ更新。存在しなければ新規挿入。

2. **Inventory の登録**
   - approved_inventory_type が UNIT の場合: InventoryUnits に 1 行追加（card_catalog_id、condition_grade＝initial_condition、storage_location_id、status＝IN_STOCK）。
   - LOT の場合: InventoryLots に 1 行追加（card_catalog_id、condition_grade、storage_location_id、qty_on_hand＝initial_qty、status＝IN_STOCK）。

3. **StockMovements への IN 記録**
   - target_type＝UNIT or LOT、target_id＝上記で登録した ID、card_catalog_id、moved_at＝現在日時、qty_delta＝initial_qty、movement_type＝IN、ref_kind＝PURCHASE または MANUAL、ref_id＝stg_id または空、operator＝ログインユーザー、notes＝「OCR 正式登録」等。

4. **OCR_Staging の更新**
   - 該当行の status を「確定」に、review_status を APPROVED に、approved_at を現在日時に、initial_qty / storage_location_id 等を記録。

**エラー時**: 在庫・StockMovements の登録をロールバック（スプシの場合は手順をトランザクション風にまとめ、失敗時は手動で整合を取る方針でも可）。OCR_Staging は「登録待ち」のままにするか、ERROR 状態を記録。

### 4.2 掲載時の在庫引当（F13）

**トリガー**: SCR-010 で新規掲載を「掲載する」実行時。

**処理**:

1. ChannelListings に 1 行追加: status＝LISTED（または DRAFT のまま API 同期後に LISTED）、list_qty、price、reserved_qty＝list_qty。
2. 対象在庫（target_type/target_id）の status を RESERVED に更新。
3. StockMovements に 1 行追加: qty_delta＝0 で movement_type＝RESERVE、ref_kind＝LISTING、ref_id＝listing_id。在庫数は変わらないが「引当済」であることを記録する方式と、qty_delta をマイナスで記録する方式のいずれかで設計する（要件では「在庫数−掲載中数量＝在庫可能数」なので、RESERVE を別カウンタで持つか、StockMovements で RESERVE をマイナス相当で表現するかを決める）。

**成約時（手動チャネル）**:

1. ChannelListings の status を SOLD、sold_at・sold_price を設定。
2. 対象在庫の status を SOLD または SHIPPED に更新。
3. StockMovements に OUT を記録: qty_delta＝−list_qty、movement_type＝OUT、ref_kind＝ORDER、ref_id＝channel_order_id（手動の場合は空でも可）。

### 4.3 在庫数・在庫可能数の算出（F9）

- **在庫数（カード種別単位）**: 当該 card_catalog_id に紐づく StockMovements の qty_delta の合計。UNIT の場合は 1 枚＝1、LOT の場合は qty_on_hand を考慮する設計（StockMovements の IN/OUT がロット数量を反映している前提）。
- **掲載中数量**: ChannelListings の status が LISTED / RESERVED の行について、list_qty（または reserved_qty）の合計（card_catalog_id 単位で集計）。
- **在庫可能数**: 在庫数 − 掲載中数量（同一 card_catalog_id で集計）。

※ Unit/Lot 単位で「どの現物が掲載中か」を管理する場合は、ChannelListings.target_id で紐づけ、在庫可能数は「status＝IN_STOCK の Unit/Lot の数または qty_on_hand の合計」でもよい。

### 4.4 Shopify 商品・在庫同期（F14）

**トリガー**: 手動「同期」ボタン、またはスケジュールバッチ。

**処理概要**:

1. 同期対象を決定: ChannelListings の channel＝SHOPIFY かつ status＝LISTED のもの、または ChannelProducts で sync_status が PENDING のもの。
2. 各対象について Shopify Admin API で Product/Variant を作成または更新。在庫数は list_qty または 在庫可能数 を反映。
3. ChannelProducts に shopify_product_id、shopify_variant_id を保存。SyncJobs に 1 件記録（status＝SUCCESS/FAILED、error_message）。
4. 失敗時は ChannelListings の sync_status＝ERROR、sync_error_message を設定。

### 4.5 Shopify 注文取り込み・在庫出庫（F15）

**トリガー**: 手動「注文取込」またはスケジュールバッチ。

**処理概要**:

1. Shopify Orders API で未処理の注文を取得。
2. 各注文の Line Item から商品（Variant ID）を特定し、ChannelProducts および CardCatalog / Inventory と照合。
3. ChannelOrders に 1 件ずつ登録。import_status＝PENDING。
4. 在庫引当: 対象在庫の status を SOLD 等に更新し、StockMovements に OUT を記録。ref_id＝channel_order_id。
5. ChannelOrders の import_status を「引当済」等に更新。SyncJobs に記録。

## 5. インターフェース仕様

### 5.1 外部連携（本システムが呼び出す）

| 対象 | 方式 | 概要 |
|------|------|------|
| Google Sheets API | REST | スプレッドシートの読み書き。本システムのデータ層。 |
| Shopify Admin API | REST | Products、Variants、Inventory、Orders の取得・更新。認証は Storefront/Admin API トークン。 |

※ Cloud Run は本システムから呼び出さない。OCR_Staging は同一スプレッドシートを共有するのみ。

### 5.2 価格取得の差し込み口

- **入力**: card_catalog_id（または serial_number）、price_value、price_type、source_name、source_type、fetched_at、その他 PriceSnapshots の項目。
- **投入先**: PriceSnapshots シートに 1 行追加。
- **呼び出し元**: 手動入力画面（SCR-014）、CSV 取込バッチ、将来の API/スクレイピング。いずれも「PriceSnapshots に 1 件投入する」インターフェースを共通化する（関数またはバッチ入力フォーマットで定義）。

### 5.3 一覧の CSV エクスポート

- **対象**: 登録待ち、カード種別、在庫、掲載、価格スナップショット等の一覧画面。
- **仕様**: 現在のフィルタ・ソート・ページ条件を反映した結果を CSV で出力する。**列ヘッダーは日本語**とする（例: カード名、セット、番号、レアリティ、在庫数、価格（円））。文字コードは UTF-8（BOM 付き推奨、Excel で開く場合のため）。日付は YYYY/MM/DD または YYYY年MM月DD日で出力する。

### 5.3 データストアアクセス

- MVP: Google Sheets API（または AppSheet 経由）で各シートを読み書き。同時編集の競合を避けるため、更新は「行の追加」「特定行の列更新」に限定し、一括更新は最小限にする。
- 将来: PostgreSQL 等に移行する際は、上記シート名をテーブル名、カラム名をそのままカラムにマッピングする。
