## 1. システム概要

### 1.1 目的

ポケモンカードの販売業者が、**スキャナーで読み取ったカードをOCRで候補化し、人が確認してシステムに登録**したうえで、**在庫・棚・複数販売チャネル（フリマ・オークション・自社EC）を一元管理**し、**Shopifyへ出品・在庫同期**および**相場の参照・値付け**までを可能にする、**カード特化型の在庫・出品管理システム**を構築する。

### 1.2 システムの位置づけ

- **本システム** = 基幹（カード台帳・在庫・棚・掲載・成約・価格履歴・ダッシュボードの正本）
- **Shopify** = クライアントが構築する自社EC。本システムから商品・在庫をAPIで同期する「販売チャネルの1つ」
- **ヤフオク・フリマアプリ** = API出品は行わず、「何をどこに何枚出しているか」「成約したか」を本システム上で管理する対象

データの正本は常に本システム側に保持し、Shopifyで売れた注文は本システムへ取り込み、在庫引当・出庫を行う。

### 1.3 対象業務フロー（概要）

1. **取り込み**: スキャナーでカードを読み取り、画像を所定の保存先（例: Google Drive）に保存する。
2. **認識**: 画像からOCR/AIでカード情報（セットコード・番号・レアリティ・名称等）を抽出し、候補として一時保存する。
3. **確認**: 人が候補を確認・修正し、OKのものだけ正式登録する。初期数量・状態・保管場所（棚番）を入力可能とする。
4. **登録**: カード種別マスタおよび現物在庫に登録し、在庫台帳に入庫記録を追加する。
5. **保管**: 棚番（ロケーション）を管理し、「どのカードがどこにあるか」を把握できるようにする。
6. **出品**: チャネル別に掲載情報を管理する。ShopifyはAPIで商品・在庫を同期し、ヤフオク・フリマは手動掲載として記録する。
7. **成約・出庫**: 成約または受注を登録し、在庫を引当・出庫する。二重販売を防ぐため状態（在庫あり/掲載中/売れ済未出荷/発送済等）を明確に管理する。
8. **相場**: 価格スナップショットを保存し、仕入・販売価格の参考とする（取得方法はAPI/手動/CSV/スクレイピング等を差し替え可能な設計）。
9. **可視化**: ダッシュボードで在庫・売上・業務・価格の指標を表示する。

### 1.4 既存コンポーネント（Cloud Run）との関係

**OCR パイプライン（画像取込〜OCR_Staging への候補登録）は、Google Cloud Run 上で動作する別サービスとして既に実装・運用されている。** 当該サービスは別リポジトリ・別管理であり、本要件定義書に基づく更新・改変・設計変更の対象外とする。

- **既存 Cloud Run の役割**
  - Google Drive の **Inbox フォルダ** に保存されたカード画像（JPEG/PNG）を検知し、ダウンロード・リサイズ後に **Gemini API** で構造化OCRを実行する。
  - 抽出結果を **同一スプレッドシート内の「OCR_Staging」シート** に追記する（同一 serial_number の場合は qty を加算して集約）。
  - 処理済み画像を **Processed** フォルダへ移動、失敗時は **Error** フォルダへ移動。リトライ上限超過時は **Fatal Error** フォルダへ移動する。
  - トリガーは HTTP 呼び出し（スケジューラ等）で実行される。
- **本システム（要件定義の対象）の役割**
  - OCR_Staging に**書き出す処理は行わない**。Cloud Run が出力した OCR_Staging を**読み取り**、登録待ち一覧の表示・確認・修正・OK/NG および正式登録（CardCatalog / Inventory / StockMovements）を行う。
  - OCR_Staging に、確認用の列（例: review_status, initial_qty, storage_location_id 等）を**管理システム側で追加・更新**することは可能とする。Cloud Run は既存の列のみを書き込む前提のため、それらの追加列に対する変更は Cloud Run 側では行わない。

本要件定義書では、上記の前提の下で「登録待ち確認」以降の機能およびデータ連携を定義する。

## 2. 用語定義


| 用語                           | 定義                                                               |
| ---------------------------- | ---------------------------------------------------------------- |
| カード種別（CardCatalog）           | セット・番号・名称・レアリティ等で一意に識別される「カードの種類」。1SKU＝1種別。                      |
| 現物在庫（Inventory）              | 実際に保有している物理的な在庫。1枚単位（InventoryUnit）または複数枚ロット（InventoryLot）で管理する。 |
| 在庫台帳（StockMovements）         | 入庫・出庫・調整・予約・解除・移動を「差分」で記録する台帳。在庫数はこの合計で算出する。                     |
| 棚番 / ロケーション（StorageLocation） | カードを保管している物理的な場所（倉庫・ゾーン・棚・ボックス・スロット等）。                           |
| チャネル（Channel）                | 販売先。SHOPIFY / YAHOO_AUCTION / MERCARI / OTHER 等。                 |
| 掲載（Listing）                  | 特定チャネルに出品している情報。API同期するもの（Shopify）と手動管理するもの（ヤフオク・フリマ）がある。        |
| OCR_Staging                  | OCR/AIが抽出したカード候補を人が確認するための一時保存領域。                                |
| 相場（PriceSnapshot）            | 外部ソースまたは手動で取得した価格情報のスナップショット。                                    |

### 2.2 表記方針

- 画面・メッセージ・マニュアルでは**日本語を優先**する。技術用語を併記する場合は「カード種別（CardCatalog）」のように日本語の後に括弧で英語を付ける。
- 略語は初出時に説明する（SKU＝最小管理単位、OCR＝光学文字認識、API＝アプリケーション連携インターフェース）。

## 3. 表示言語・ロケール（日本語前提）

本システムは**日本人利用者向け**であり、以下を満たす。

- **表示言語**: システムの画面に表示するすべてのテキスト（ラベル、ボタン、メッセージ、ヘルプ、エラー、プレースホルダ、確認ダイアログ）は**日本語**とする。英語のみの画面やメッセージは出さない。
- **日付・時刻**: 表示形式は日本向けとする。例: **YYYY年MM月DD日**、**YYYY/MM/DD**、時刻は **HH:mm**（24時間制）。和暦（令和X年）表示は任意で対応可能とする。
- **数値**: 桁区切りは**カンマ**（例: 1,234）。小数点は**ピリオド**。
- **通貨**: デフォルトは**日本円（円、JPY）**。価格表示では「円」または「¥」を用いる。
- **エラーメッセージ**: 利用者に表示するエラー・バリデーション文言は**日本語**で、原因と対処が分かりやすい表現とする。技術的なエラーコードをそのまま表示する場合は、その前に日本語の説明を付ける。
- **コード・API・DB**: 内部の変数名・API パス・テーブル名・列名は英語のままでもよい。**利用者が直接目にする部分のみ**日本語を担保する。

## 4. スコープ

### 4.1 対象範囲（本システムで実現する機能）

- **画像取込・OCR候補生成**: これらは**既存の Cloud Run サービス**が担当する。本システムは、Cloud Run が書き込んだ **OCR_Staging を入力**として利用する。Cloud Run の改修・設計変更は本要件の対象外とする。
- 登録待ち候補の一覧表示・確認・修正・OK/NG 判定（OCR_Staging の参照・更新）
- カード種別マスタ（Sets, CardCatalog）の登録・更新・参照
- 現物在庫（InventoryUnits / InventoryLots）の登録・更新・参照
- 在庫移動（StockMovements）の記録と在庫数の算出（差分台帳方式）
- 棚番（StorageLocations）のマスタ管理と在庫との紐づけ
- チャネル別掲載（ChannelListings）の登録・状態更新・成約記録
- Shopify との連携（商品作成/更新、在庫同期、注文取り込み）
- 外部チャネル（ヤフオク・フリマ）の掲載・成約の「管理」（手動登録）
- 価格スナップショット（PriceSnapshots）の保存・参照と、取得処理の差し込み口（API/手動/CSV/スクレイピング等）
- ダッシュボード（在庫・販売・業務・価格の主要指標の表示）
- 上記を支えるデータ保存（MVP期はスプレッドシート可、スキーマはDB移行前提）
- 一覧画面からの**CSV エクスポート**（在庫・掲載・カード種別・価格等の一覧を日本語ヘッダで出力し、表計算や棚卸に利用できるようにする）

### 4.2 対象外（本要件定義の範囲外）

- **Cloud Run サービス本体の更新・改変・設計**（画像取込・Drive 操作・Gemini 呼び出し・OCR_Staging への追記ロジックは別管理）
- スキャナー機器の制御・ドライバ（画像が所定場所に保存される前提）
- Shopify ストアの構築・テーマ編集・決済設定（クライアント側で実施）
- ヤフオク・フリマへの自動出品・自動落札（管理のみ）
- 決済・会計・経理システムとの直接連携（将来要件として検討可能）
- 実店舗POS（将来拡張として検討可能）
- スマートフォン専用アプリの開発（必要に応じてWebのレスポンシブで対応）

### 5.2 機能要件詳細

#### F1–F2: 画像取込・OCR候補生成（既存 Cloud Run が担当・本要件の変更対象外）

- **F1・F2 は既存の Cloud Run サービスで実装済みである。** 本要件定義書では当該サービスの改修・設計は行わない。
- Cloud Run の動作概要（参考）:
  - **入力**: Google Drive の **Inbox** フォルダに保存されたカード画像（JPEG/PNG）。画像はリサイズ（最大辺 1024px）後に **Gemini API**（構造化JSON出力）で解析される。
  - **出力**: 同一スプレッドシート内の **OCR_Staging** シートに行を追記。同一 serial_number の場合は既存行の qty を加算して集約する。
  - **画像の移動**: 処理成功時は **Processed** フォルダへ、失敗時は **Error** フォルダへ。リトライ上限（例: 3回）超過時は **Fatal Error** フォルダへ移動する。再処理用に `/retry` エンドポイントで Error から Inbox へ戻す運用が可能。
- 本システム（管理画面・登録処理）は、**Cloud Run が書き込んだ OCR_Staging の内容を読み取り、確認・正式登録に利用する**。

#### F3–F4: 登録待ち確認・正式登録

- **F3**: 登録待ち一覧画面で、OCR_Staging の候補を一覧表示する。各候補について、元画像・左下サムネイル・抽出された項目を表示し、作業者が内容を修正できるようにする。OK/NG/要再スキャンのいずれかを選択できるようにする。初期数量・状態（コンディション）・保管場所（棚番）・ intended_channels（出品予定チャネル）等を入力可能にできるとよい。
- **F4**: OK が押された候補について、以下を実行する。
  - CardCatalog に該当カード種別がなければ登録し、card_id / serial_number を生成する。既にあれば更新しないか、必要に応じて更新する。
  - InventoryUnit または InventoryLot を、approved_inventory_type および initial_qty に従って登録する。
  - StockMovements に IN の移動を initial_qty 分追加する。
  - OCR_Staging の該当行を「確定」等の状態に更新する。
- NG または要再スキャンの場合は、状態を更新するのみとし、在庫には反映しない。

#### F5–F6: セット・カード種別マスタ

- **F5**: Sets には set_code, set_name_ja, series, release_date, total_cards, regulation_set 等を保持する。参照・登録・更新が可能であること。一括取込（CSV等）は任意。
- **F6**: CardCatalog には serial_number, set_code, card_number, number_total, name_ja, regulation_mark, rarity, card_type, trainer_subtype, poke_type, art_variant, holo, language, image_ref_standard, searchable_text 等を保持する。検索・参照・登録・更新が可能であること。在庫数は CardCatalog には持たず、StockMovements 等から算出して表示する。

#### F7–F9: 現物在庫・在庫移動・在庫数算出

- **F7**: InventoryUnits（1枚単位）は condition_grade, storage_location_id, status, front_image_url, back_image_url, acquisition_cost 等を管理する。InventoryLots（ロット）は qty_on_hand, condition_grade, storage_location_id, status, lot_type 等を管理する。登録・更新・参照が可能であること。status は IN_STOCK / RESERVED / LISTED / SOLD / SHIPPED / HOLD 等を想定する。
- **F8**: StockMovements には target_type（UNIT/LOT）, target_id, card_catalog_id, moved_at, qty_delta, movement_type（IN/OUT/ADJUST/RESERVE/RELEASE/TRANSFER）, ref_kind, ref_id, operator, notes を記録する。在庫数はこのテーブルの qty_delta の合計で算出する（直接在庫数を上書きしない）。
- **F9**: 画面上で「在庫数」「在庫可能数」（在庫数−掲載中数量等）を表示する。必要に応じてカード種別単位・現物単位・棚別で集計できるようにする。

#### F10–F11: 棚番・在庫と棚の紐づけ

- **F10**: StorageLocations に warehouse, zone, shelf, rack, bin, slot, barcode, active_flag 等を保持する。階層的に管理し、参照・登録・更新が可能であること。
- **F11**: InventoryUnits / InventoryLots に storage_location_id を保持し、棚別の在庫一覧・検索が可能であること。

#### F12–F16: チャネル掲載・引当・Shopify連携・手動チャネル

- **F12**: ChannelListings に channel（SHOPIFY / YAHOO_AUCTION / MERCARI / OTHER）, listing_mode（API_SYNC / MANUAL_MANAGED）, target_type / target_id（Unit/Lot）, card_catalog_id, external_listing_id, status（DRAFT/LISTED/RESERVED/SOLD/ENDED/SYNC_ERROR）, list_qty, price, reserved_qty, listing_title, listing_image_urls, published_at, ended_at, sync_status, sync_error_message 等を保持する。登録・更新・一覧・検索が可能であること。
- **F13**: 掲載登録時に、該当在庫の status を RESERVED 等に更新し、StockMovements に RESERVE を記録する。成約時は SOLD 等に更新し、OUT を記録する。これにより他チャネルとの二重販売を防ぐ。
- **F14**: Shopify 連携では、本システムで管理する対象商品・在庫を Shopify API を用いて Product / Variant として作成・更新し、在庫数（Inventory）を同期する。ChannelProducts（channel, card_id, shopify_product_id, shopify_variant_id, sync_status, last_synced_at）を保持し、同期状態を管理する。
- **F15**: Shopify の注文を API で取得し、ChannelOrders に取り込む。取り込んだ注文に紐づく在庫を引当（RESERVE→OUT）し、本システムの在庫数を減らす。発送済みは SHIPPED 等で記録する。
- **F16**: ヤフオク・フリマについては、作業者が手動で「掲載した」「成約した」を ChannelListings に登録・更新する。成約登録時に在庫引当・出庫（StockMovements に OUT）を行う。

#### F17–F18: 価格スナップショット・取得差し込み口

- **F17**: PriceSnapshots に card_catalog_id, source_name, source_type（API/SCRAPING/MANUAL/CSV）, condition_assumption, price_type（ASK/SOLD/LOWEST/MEDIAN）, price_value, currency, fetched_at, source_url, raw_payload 等を保持する。参照・一覧・時系列表示が可能であること。
- **F18**: 価格データの取得は、API・手動入力・CSV取込・スクレイピング等、複数の方法を差し替え可能な設計とする。取得処理は「PriceSnapshots に投入する」インターフェースを共通化する。

#### F19–F22: ダッシュボード

- **F19**: 在庫ダッシュボードで、総在庫数・在庫金額（評価額）・出品可能数・チャネル別出品数・棚別在庫数・登録待ち件数 を表示する。
- **F20**: 販売ダッシュボードで、日次/週次/月次売上・粗利・チャネル別売上・商品回転率・滞留在庫ランキング を表示する。
- **F21**: 業務ダッシュボードで、OCR成功率・人手確認待ち数・連携エラー数・棚卸差異・二重販売リスク件数 を表示する。
- **F22**: 価格ダッシュボードで、相場推移・自社価格との差・値付け見直し対象・高滞留カード一覧 を表示する。
- 集計期間の指定（日・週・月）および必要に応じたフィルタ（チャネル・棚・カード種別等）が可能であること。

#### F23–F24: 連携ログ・監査ログ

- **F23**: SyncJobs に連携種別（例: Shopify商品同期・在庫同期・注文取込）、実行日時、成否、エラー内容、処理件数 等を記録し、一覧・検索が可能であること。
- **F24**: 重要なデータ登録・更新について、操作者・日時を記録する機能は、MVP では任意とし、将来拡張で対応する。

## 7. 非機能要件

### 6.1 性能

- 登録待ち一覧の表示は、通常利用想定件数（例: 数百〜数千件）において、許容できる応答時間（例: 5秒以内）で表示できること。
- 在庫数・在庫可能数の算出は、StockMovements の集計により行うため、データ量が増えた場合でも集計が完了するよう、インデックス設計や集計キャッシュを検討する。
- Shopify API の呼び出しは、レート制限に従い、必要に応じてバッチ・キューで実行する。

### 6.2 可用性・運用

- MVP 期はスプレッドシートをバックエンドとする場合、同時編集の競合に注意し、必要に応じてロックや更新頻度のガイドラインを設ける。
- 本番運用時（DB移行後）は、バックアップとリストア手順を定め、定期的なバックアップを取得する。
- 連携エラーが発生した場合、SyncJobs に記録し、管理者が確認・再実行できるようにする。

### 6.3 セキュリティ

- カード画像・在庫・価格・売上データは機密情報として扱い、アクセスは認証された利用者のみに限定する。
- Shopify API キー・トークン等は、環境変数または秘密情報管理で保持し、ソースコードにハードコードしない。
- 外部サイトからの相場取得（スクレイピング）を行う場合は、利用規約・法的要件を遵守し、過負荷をかけないよう配慮する。

### 6.4 保守性・拡張性

- データモデル（テーブル・シート構成）は、将来の RDB（例: PostgreSQL）への移行を前提に設計する。
- チャネル追加（新規フリマ・オークション等）や価格取得方法の追加が、既存設計を大きく壊さずに行えるようにする。
- 主要なビジネスルール（在庫数算出、引当ルール）はドキュメント化し、変更時に追跡可能にする。

- 管理画面は Web ブラウザで利用することを想定する。AppSheet または自前の軽量 Web のいずれでも、上記機能要件を満たすこと。

- 在庫一覧・掲載一覧では、検索・フィルタ・ソートが可能であること。
- ダッシュボードは、主要指標が一覧で把握できるレイアウトとする。
- 一覧画面では、現在のフィルタ・ソート条件を反映した CSV エクスポート（日本語ヘッダ）を提供する。

## 10. 制約・前提条件

- **OCR パイプライン（Drive Inbox → Gemini → OCR_Staging 追記・画像移動）は既存の Cloud Run サービスが担当する。** 当該サービスの更新・改変・設計は本要件定義の対象外であり、別途管理する。本システムは OCR_Staging の「読取」および確認用列の「更新」のみを行う。
- スキャナーから画像が Google Drive の **Inbox** に保存される運用は既存のとおりとする。処理済み画像は Cloud Run により Processed / Error / Fatal Error へ移動される。
- Shopify ストアはクライアントが自ら構築・運用する。本システムは同一テナントの API を使用して商品・在庫・注文を連携する。
- ヤフオク・フリマへの出品は手動で行い、本システムでは「掲載した」「成約した」の記録と在庫引当・出庫のみを行う。
- MVP 期のデータ保存はスプレッドシート（Google Sheets 等）でもよいが、スキーマ（シート名・列名・データ型の考え方）は将来の RDB 移行を前提に設計する。OCR_Staging の列構成は、Cloud Run が出力する列を前提とし、管理システム側で追加する列は Cloud Run に影響させない。
- 相場取得でスクレイピングを行う場合は、各サイトの利用規約および関連法規を遵守する。設計上は取得方法を差し替え可能にし、API や CSV に切り替え可能にする。

## 11. 受入条件の考え方

- **登録フロー**: 既存 Cloud Run により、Inbox の画像から OCR 候補が OCR_Staging に登録されること。本システムの登録待ち一覧にその内容が表示され、作業者が OK した候補が CardCatalog / Inventory / StockMovements に正しく反映され、在庫数が増えること。
- **在庫**: 入庫・出庫・調整を記録した結果、在庫数および在庫可能数が正しく算出・表示されること。掲載時に在庫が引当され、成約・出庫で減ること。
- **棚**: 棚番マスタを登録し、在庫に保管場所を紐づけたうえで、棚別の在庫一覧が参照できること。
- **Shopify**: 本システムで指定した商品・在庫が Shopify に反映され、Shopify で受注した注文が本システムに取り込まれ、在庫が減ること。
- **手動チャネル**: ヤフオク・フリマの掲載・成約を手動登録し、在庫が引当・出庫されること。
- **ダッシュボード**: 在庫・販売・業務・価格の各指標が、定義した集計条件で表示されること。
- **非機能**: 想定データ量・同時利用で、要件で定めた性能・運用・セキュリティの目安を満たすこと。
